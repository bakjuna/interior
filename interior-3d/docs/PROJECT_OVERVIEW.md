# interior-3d — 프로젝트 종합 문서 (AI 친화)

> 이 문서는 향후 AI(Claude 등)가 이 저장소에 처음 들어왔을 때 **즉시 맥락을 파악**하고
> 작업을 이어갈 수 있도록 작성되었다. 코드 인용보다는 **개념·좌표계·규약·작업 절차**를
> 명시적으로 기록한다. 코드 자체보다 *왜 그렇게 짜여 있는가*가 핵심이다.

---

## 0. 한 줄 요약

실제 평면도(은행현대 84㎡)를 mm 단위로 측정해 TypeScript 데이터로 옮기고,
React Three Fiber로 **평면도 / 조감도 / 1인칭 워크스루** 세 가지 뷰를 한 모델에서
렌더링하는 인테리어 시뮬레이터.

## 1. 핵심 테마

1. **단일 진실 소스(Single Source of Truth)**
   - 모든 형상은 `src/data/apartment.ts` 한 파일의 숫자에서 파생된다.
   - `ApartmentModel.tsx`는 이 데이터를 먹고 mesh로 토해낼 뿐, 형상 정보를 자체 보관하지 않는다.
   - → 평면도(2D)와 3D 모델이 **항상 자동으로 동기화**된다.

2. **mm 단위 정밀도 + m 단위 좌표계**
   - 입력은 항상 실측 mm (`4220mm`, `1607mm` …) 를 그대로 주석에 남기고,
     코드에는 m 단위 (`4.220`, `1.607`)로 환산.
   - 작은 오차도 누적되면 벽이 어긋나기 때문에, 모든 위치는 *기준점에서의 누적식*으로 표기.

3. **벽은 "선분 + 두께"**
   - 벽 = `start: [x,z]` → `end: [x,z]` + `thickness`. height/bottomY가 비면 전체 높이 벽.
   - 창/문은 같은 선분 위에서 `height`/`bottomY`로 잘라낸 **여러 개의 부분 벽**으로 표현.
   - → 평면도, 3D, 워크스루가 동일한 wall 배열을 공유한다.

4. **"방"은 시각적 표시일 뿐, 충돌·조명의 단위**
   - 바닥 색/타일/단내림은 `Room`이 들고 있고, 충돌은 워크스루의 별도 로직이 처리.
   - 조명은 `downlightGroups`가 `bounds`로 영역을 정의 → 워크스루 카메라의 현재 위치가
     어떤 그룹에 속하는지로 ON/OFF.

5. **세 가지 뷰, 한 모델**
   - `/` 평면도 (Orthographic top-down, 라벨/치수 표시)
   - `/birdseye` 조감도 (Perspective + WASD 자유 비행, 천장 OFF)
   - `/walkthrough` 1인칭 (Pointer Lock, 충돌, 야간/주간 토글, 도어 인터랙션)

## 2. 좌표계 규약 (★ 가장 중요)

```
            -Z  (북, 화면 위, 침실/세탁실/주방 쪽)
             ↑
   -X ←──────┼──────→ +X
             ↓        (서: 안방,   동: 거실/현관)
            +Z  (남, 화면 아래, 발코니 쪽)
```

- **X = 가로(동서), Z = 세로(남북), Y = 높이(위아래)**
- Three.js 기본 좌표계와 동일. 평면도에서는 Y를 무시하고 X-Z 평면을 위에서 본다.
- **+Z가 화면 아래(남쪽 발코니)**, -Z가 화면 위(북쪽 침실)로 고정되어 있다. 한 번 뒤집은
  적이 있다 → `docs/floor_plan001.md` 참고. 다시 뒤집지 말 것.
- 모든 좌표 원점은 **거실 좌상단 내측 코너 (0, 0)**.
  - 거실 가로 = `LR_W` = 3.972m
  - 거실/안방 세로 = `LR_D` = 3.666m
  - 안방은 거실 왼쪽, 공유벽 두께(`WALL_THICKNESS = 0.2`)만큼 음수 X에서 시작.
- 벽 두께 절반은 항상 `T2 = 0.1`. 벽 중심선 좌표를 쓸 때마다 등장한다.
- "내측면(inside face)" / "외측면(outside face)" 용어를 코드 주석에서 엄격히 구분한다.
  치수가 어긋나면 거의 항상 이 둘을 헷갈린 것이다.

### 자주 쓰는 파생 상수 (apartment.ts 상단)

| 이름 | 의미 |
|---|---|
| `mbLeft` | 안방 좌측 내측면 X |
| `mbDoorHinge` / `mbDoorEnd` | 안방 상단 도어 경첩/끝 X |
| `mbBath*` | 안방 욕실 내측 4변 |
| `bath2*` | 메인욕실 좌표 |
| `baby*` | 아기방 좌표 |
| `stair*`, `right*` | 세탁실 계단형 벽 꺾임점 |
| `verandaTop/Bottom/WallEnd` | 메인 베란다 Z |

새 공간을 추가할 때는 **반드시 기존 상수를 기준으로 누적식**을 쓴다.
하드코딩된 절대 좌표를 새로 만들지 말 것 → 한 곳을 바꾸면 도미노가 깨진다.

## 3. 데이터 모델

`src/data/apartment.ts` 가 전부다 (~650줄).

```ts
interface Wall {
  start: [number, number]   // [x, z]
  end:   [number, number]
  thickness: number
  height?: number           // 부분 벽이면 지정 (창/문 위·아래)
  bottomY?: number          // 부분 벽 하단 Y
  isBathroom?: boolean      // 타일 머티리얼
  isBalcony?: boolean       // 콘크리트 머티리얼
}

interface Opening {          // doors / windows 공통
  position: [number, number] // 벽 위의 한 점 (벽 중심선)
  width: number
  height: number
  sillHeight: number         // 0이면 도어
  axis: 'x' | 'z'            // 벽이 X축이면 'x'
  type: 'door' | 'window'
}

interface Room {
  name: string
  center: [number, number]
  size:   [number, number]
  color: string
  floorY?: number            // -0.03이면 단내림(욕실/현관)
  floorTile?: 'porcelain' | 'entrance'
  tileSize?: number          // m. 미지정 시 강마루
}

interface Closet { name; position:[x,y,z]; size:[w,h,d]; color }
```

배열들:

- `walls: Wall[]` — 벽 + 창/문 위아래의 부분벽 + 상인방(lintel) + 단차벽
- `doors: Opening[]` / `windows: Opening[]` — 평면도 라벨링 + 3D 구멍 위치
- `doorArcs[]` — 평면도에 그리는 도어 회전 호 (경첩/방향/거울)
- `rooms: Room[]` — 바닥 색/타일/단내림
- `closets: Closet[]` — 붙박이장 (호두무늬 도어 텍스처)
- `downlightGroups: DownlightGroup[]` — 방별 조명 클러스터 + 점유 영역
- `downlights` — 위 그룹들을 평탄화한 것 (평면도/조감도용)

### 절대 추가하지 말 것
- 형상 데이터를 컴포넌트 안에 하드코딩 (벽·창·도어 모두 여기에 추가)
- 불필요한 helper / 추상화 (이 파일은 의도적으로 *납작한 데이터 파일*이다)

## 4. 컴포넌트 구조

```
src/
  routes/
    __root.tsx       — 공통 레이아웃 + 네비
    index.tsx        — / → FloorPlanView
    birdseye.tsx     — /birdseye → BirdsEyeView
    walkthrough.tsx  — /walkthrough → WalkthroughView
  components/
    ApartmentModel.tsx   (~3500줄)  ★ 모든 3D 메시 생성기
    FloorPlanView.tsx    (~510줄)   2D 평면도 (Orthographic)
    BirdsEyeView.tsx     (~120줄)   3D 조감도 (Orbit + WASD)
    WalkthroughView.tsx  (~605줄)   1인칭 (PointerLock + 충돌)
  data/
    apartment.ts         (~650줄)
```

### 4.1 ApartmentModel.tsx

- 입력 props: `showCeiling`, `playerPos`, `isNight`, `allLightsOn`, `showCityBackground`
- 텍스처 로드: 호두 마루, 포세린 타일, 현관 타일, 호두 도어, 실크 벽지, 도시 배경(낮/밤×북/남)
- `useMemo`로 한 번 만들고 끝나는 메시들:
  - 방 바닥 (room.floorTile에 따라 텍스처 분기)
  - 벽 (`walls.map`, isH/isV에 따라 boxGeometry 회전)
  - 창문 (프레임/멀리언/유리 패널 — 문/창 다 직접 조립)
  - 붙박이장 (`closets`, 도어 분할 600mm 기준 자동 계산)
  - 조명 (다운라이트 = SpotLight + 디스크 메시, 그룹별 토글)
- `playerPos`로 현재 점유 그룹을 찾고, **그 그룹의 조명만 ON**. 나머지는 OFF.
  → 야간 워크스루의 핵심 연출.
- gltf/glb 로더로 침대, 변기, 세면대, 부엌 모델 로드 (`public/models/`).

### 4.2 FloorPlanView.tsx

- `OrthographicCamera` + `MapControls` (회전 잠금, 패닝만 허용)
- `up = [0, 0, -1]` → 화면 위가 -Z 방향(북쪽)이 되도록 설정
- 방 바닥 plane → 벽 라인 → 창/문 표시 → 도어 호(arc) → 라벨/치수
- "맹지" 영역은 별도의 빗금으로 표시

### 4.3 WalkthroughView.tsx

- Pointer Lock + WASD + 모바일 가상 조이스틱
- **한글 입력 상태에서도 동작**하는 키 매핑 테이블 (`KO_TO_EN`)
- 충돌 처리: 카메라가 벽 안쪽 0.2m 마진을 가지도록 슬라이딩
- 도어 인터랙션: `doorArcs` 기반으로 클릭 시 회전 보간
- 야간/주간 토글 → ApartmentModel의 `isNight` prop, 다운라이트 그룹 ON/OFF + 도시 배경 야경

### 4.4 BirdsEyeView.tsx

- `OrbitControls` + 자유 비행 WASD (앞/뒤/좌/우, 카메라 forward 기반)
- `showCeiling=false`로 천장 제거 → 위에서 내려다보기

## 5. 인테리어 자체 구성 (실제 도면)

### 5.1 호수 정보
- **은행현대 84㎡** (전용 84제곱미터, 한국 표준 국민평형 "34평")

### 5.3 공간별 상세

#### 거실 (3.97 × 3.67m)
- 남측 거실 창: 폭 2000 × 높이 2000mm 통창 (sill 0)
- 호두 마루(walnut floor), 천장 다운라이트 8개(4코너 × 2)
- **ㄱ자 패브릭 소파**: 거실 창문 앞, 창에서 100mm 띄움. gltf 모델(`/models/scene.gltf`) 사용
- **벽걸이 TV**: 거실/현관 사이 1481mm 벽 (1450 × 840mm, 65인치 상당)
- 안방과의 공유벽(거실 쪽)에 **수납장(450mm 깊이 × 3.66m)** — 호두 무늬 도어
- 거실/복도 공유벽은 800mm 폭만 세우고 **하단 100mm를 띄워 코브 LED 간접조명** → 떠 있는 벽 효과

#### 안방 (4.22 × 3.67m)
- 좌측에 **붙박이장 550mm × 약 3.0m** (상단까지 풀높이, 호두 도어). 첫 600mm는 화장대 자리로 비움
- **가벽(50mm 두께)** 으로 침대 영역과 화장대/통로 영역을 가벼운 분리 → 드레스룸 느낌
- 가벽에 1066mm 개구부 → 안방욕실 동선 확보
- 화장대 거울 4변 간접조명 (상/하/좌/우 LED 라인)
- 남측 창: 폭 2000 × 높이 750mm (sill 1000mm) — 침대 헤드 위 가로 띠창
- 안방 다운라이트는 특수 케이스: 가벽 때문에 3컬럼 × 4행 = 12개

#### 안방 욕실 (마스터 욕실, mb_bath)
- 포세린 타일 300mm, 단내림 -30mm
- **변기 + 세면대 + 욕조(또는 샤워)** — gltf `Toilet`, `Sink` 모델 배치
- 좌측벽 부착 원형 거울 + 후면 간접조명 (메인등 연동)
- 안방에서 가벽 우회로 직접 진입

#### 아기방 (3.06 × 2.71m)
- 북측 창: 1996 × 710mm (sill 1040), 채광 + 거리뷰
- 좌측에 수납장 450mm (높이 풀)
- **일룸 쿠시노**: 범퍼 포함 1115 × 2065mm, 헤드를 동측 문측 벽에 붙이고 우측을 그대로 보임.
- 출입문은 아래로, 손잡이 220mm 오프셋

#### 메인 욕실 (2173 × 1413mm 내측, 복도 진입)
- 단내림 -30mm, 포세린 300타일
- **샤워부스 + 변기 + 세면대 + 벽 니치(niche)** 4종 세트 — 한국형 3분할 욕실의 정석
- 깊이 950mm 샤워부스(유리벽 + 유리문) → 건식/습식 분리
- 벽 니치(샴푸/바디워시 수납) 음각으로 표현
- 북쪽 벽 부착 원형 거울 + 후면 간접조명

#### 주방 (2.55m × 약 3.5m, 11자 + 식탁)
- 좌측 하부장 위 **월넛 오픈 선반 2단** (선반 길이 = 벽 길이의 80%, 선반 깊이 250mm, Y=1.28/1.68m)
- ㄱ자 라인 조명(주방 천장 매립) — 일반 다운라이트 대신
- **4도어 양문형 냉장고 (915 × 920 × 1800mm)** + **김치냉장고 (810 × 790 × 1800mm)** 빌트인 — 상부장과 측면 필러로 면 일치 마감
- **싱크대(800mm) + 2도어 하부장** + 인덕션/오븐 라인
- **식탁 1500 × 800mm**: 대리석 상판 + 가운데 나무 다리(센터 페디스털)
- 식탁 옆 카운터에 **쿠쿠 밥솥(`RiceCooker` gltf)** + **광파오븐(`LightWaveOven`)** + **쿠쿠 정수기(`CuckooWaterPurifier`)** 직접 모델링

#### 작업실 (동측, 부부 공동 워크스페이스)

## 5b. 디자인 평가

> 모델하우스가 아니라 *실제로 살 사람이 직접 그려본 도면*이라는 관점에서, 잘 된 점과
> 개선할 점을 분리해 평가한다. 디자이너가 다음 작업을 받았을 때 어디부터 손볼지 알 수
> 있도록 정리.

### 5b.1 잘 된 점 (Strong points)

1. **국평 트렌드의 명료한 재현 — 월넛 우드 + 화이트 + 그레이 3톤**
   호두 마루 + 호두 도어 + 흰 벽지 + 그레이 포세린이라는 가장 안정적인 한국 국평 톤을
   고집스럽게 끝까지 유지했다. 톤이 찢어지지 않아 어떤 컷을 잘라도 한 집이라는 인상이
   유지된다.

2. **간접조명의 적극 활용 — 면 발광 위주**
   - 거실/복도 공유벽 일부는 *하단 100mm를 띄워서* 벽이 떠 보이게 처리
   - 안방/아기방 단내림 라인 코브
   - 화장대/욕실 거울 4변 RectAreaLight
   - 신발장 오픈 선반 상부 라인
   다운라이트만으로 처리한 평범한 천장 조명 대신, *벽과 가구 자체를 발광체로* 만든
   접근이 일관되게 깔려있다. 야간 워크스루에서 가장 큰 임팩트를 만든다.

3. **부부 동수 워크스페이스 (작업실 2인 책상 + 안방 화장대)**
   재택 비중이 큰 30~40대 부부의 라이프스타일 가정에 잘 맞춘다. 작업실 책상 두 개를
   *나란히가 아니라 마주보지 않게 같은 방향으로* 배치한 점이 현실적이다 (집중 시 시선
   충돌 방지).

4. **주방의 빌트인 디테일이 세밀**
   냉장고/김치냉장고를 단순히 박스로 두지 않고 **상부장 + 측면 필러로 면 일치 마감**까지
   재현했다. 또 식탁이 별도 다이닝룸이 아니라 *주방 11자 라인의 연장선*에 붙어 있어서
   84㎡의 한정된 면적을 효율적으로 쓴다. 월넛 오픈 선반(2단)이 차가운 화이트 주방에
   온기를 더하는 포컬 포인트가 된다.

5. **새장이라는 캐릭터 공간**
   메인 베란다 일부(1340mm)를 **루버 도어로 분리한 새장**으로 만든 건 모델하우스에서는
   절대 안 나오는 *이 집에 사는 사람의 취향 흔적*이다. 이런 디테일 하나가 평면도를
   "사람이 사는 집"으로 만든다. 디자인적으로 가장 칭찬할 만한 부분.

6. **욕실 분리 전략 — 마스터 / 공용**
   안방 욕실(욕조형) + 메인 욕실(샤워부스 + 니치형) 두 곳을 명확히 다른 캐릭터로 분리.
   같은 마감재인데도 한쪽은 휴식, 한쪽은 효율로 차별화.

7. **중문 + 모루유리**
   현관 중문에 모루유리를 씀으로써 *프라이버시는 지키되 빛은 통과*시키는 한국 아파트의
   클래식한 해법을 충실히 반영. 신발장과의 연결도 깔끔.

### 5b.2 아쉬운 점 / 리스크 (Weak points)

1. **포컬 포인트(아트월)가 약하다**
   거실 TV 벽이 단순한 흰 실크벽지 + 벽걸이 TV. 요즘 트렌드는 TV 벽에 **마이크로시멘트,
   세라믹 슬랩, 우드 패널, 또는 매립 가구장** 중 하나를 쓰는데, 여기는 비어 있다.
   집의 첫인상을 결정하는 면이 가장 평범하다. → 거실 TV 뒤쪽을 *호두 패널 슬랫* 또는
   *세라믹 슬랩*으로 마감하면 즉시 격이 올라간다.

2. **소재 종류가 너무 적어서 단조로움 위험**
   호두 + 흰 벽지 + 포세린 + 대리석(주방 상판)으로 끝. 거실/주방/안방이 다 비슷한
   톤이라 *공간 간 대비*가 부족하다. 예: 안방 침실 한 면에 진한 그린/네이비 페인트
   포인트 벽, 또는 **패브릭 헤드보드 벽** 정도가 들어가면 안방의 캐릭터가 살아난다.

3. **천장 디자인이 평면적**
   천장이 단일 평면 + 다운라이트만 있다. 거실 *우물천장(coffered ceiling)* 또는 *간접
   조명을 두른 단내림 박스*가 거의 모든 한국 신축 인테리어의 기본인데 빠져 있다. 거실의
   3.6m × 3.97m 박스 형태가 그대로 노출되어 평평해 보인다.

4. **소파/식탁/침대 외 데코 요소가 비어 있음**
   - 거실: 러그, 사이드테이블, 플로어램프, 식물, 액자 → 없음
   - 안방: 협탁, 무드등, 침구 디테일 → 없음
   - 아기방: 장난감 수납, 책장, 벽 데코 → 없음
   3D 모델로 재현하는 건 비싸지만, *최소한 거실 러그 1장과 플로어램프 1개*만 추가해도
   "사람이 사는 집"의 설득력이 크게 올라간다. 지금은 모델하우스 직전 단계 같다.

5. **창문 처리(커튼/블라인드) 부재**
   모든 창이 노출된 유리. 안방/아기방은 거실과 달리 *프라이버시와 암막*이 필수인데
   커튼 박스/블라인드가 없다. 이건 디자인이라기보다 *기능 누락*에 가깝다.

6. **안방 동선 — 가벽 드레스룸의 폭**
   드레스룸/가벽 분리는 좋지만, 1066mm 개구부 + 가벽 50mm 두께만으로는 *지나가는 통로*
   인지 *드레스룸 영역*인지 모호하다. 가벽 안쪽에 화장대 + 거울이 있으니 결국 *통로 겸
   파우더 공간*인데, 폭이 좁아 화장대 의자를 놓으면 동선이 막힌다. → 가벽을 100mm
   더 우측으로 빼든지, 화장대를 코너형으로 바꾸는 게 낫다.

7. **세탁실의 L자 동선이 과해 보임**
   계단형 4단 꺾임 벽은 *건축적 제약*이 그대로 반영된 결과로 보이는데, 워시타워 외에
   별다른 가구가 없어 공간 효율이 낮다. 빨래 건조대, 다용도 선반, 청소도구 수납 정도는
   더 들어가야 84㎡의 귀한 면적이 살아난다.

8. **식탁의 위치**
   식탁이 주방 11자 라인 옆에 붙어 있는데, *식탁 한쪽이 주방 동선과 겹친다*. 4인 가족이
   식탁에 앉아 있을 때 누군가 냉장고를 열려면 등 뒤를 지나야 하는 구조. 한국 국평에서
   흔한 문제이지만, 여기서는 *식탁을 90도 돌려 거실과 주방의 경계*로 쓰는 편이 더
   여유롭다.

9. **컬러 액센트 부재**
   톤이 너무 안전하다. 모델하우스적 안정감이지만, *기억에 남는 한 장면*이 없다. 욕실
   거울 백라이트의 따뜻한 톤(`#fff5e6`) 정도가 유일한 색감 액센트. 새장의 루버 도어를
   웜 그린 또는 머스타드로 칠하기만 해도 새장이 즉시 *집의 시그니처*가 된다.

10. **외부 뷰가 대형 plane 한 장**
    북/남 도시 뷰가 *140 × 70m 대형 plane 1장*. 거리감이 없어서 모든 창에서 같은 거리에
    같은 풍경이 보인다. 패럴랙스(거리별 빌딩 깊이)가 없으면 야경이 평면적으로 느껴진다.
    → 빌딩을 3~4단 depth로 분할하거나, 단순한 box city를 추가하면 워크스루의 *바깥 풍경*
    이 살아난다.

### 5b.3 한 줄 총평

**"국평 인테리어의 모범답안을 정확하고 차분하게 그렸다. 다만 답안에서 한 발 더 나간
*이 집만의 시그니처*는 새장 하나뿐이다."**

다음 작업을 받는다면 우선순위는:
1. 거실 TV 벽 아트월 (가장 큰 ROI)
2. 거실 러그 + 플로어램프 + 협탁 (사람 사는 집 만들기)
3. 안방 헤드보드 벽 + 침구 (안방 캐릭터)
4. 천장 우물 + 간접조명 박스 (공간감 확장)
5. 커튼/블라인드 (기능 + 패브릭 텍스처)
6. 새장 도어 컬러 액센트 (시그니처 강화)

## 6. 클로드 코드를 어떻게 활용했는가 (방법론) ★

> 이 섹션이 미래의 AI에게 가장 가치 있는 부분이다. **"도면 → 3D"를 한 번에 시키지 않는다.**
> 작은 단위로 쪼개서 *대화 한 턴 = 벽 한 면 / 공간 하나*가 되도록 진행한다.

### 6.1 절대 하지 않은 것

- ❌ "이 도면 사진을 보고 전체 평면도를 코드로 만들어줘"
- ❌ "안방, 거실, 아기방, 욕실, 주방 다 한 번에 추가해줘"
- ❌ 실측치 없이 비율로 짐작하기
- ❌ 좌표를 절대값으로 새로 만들기

### 6.2 실제로 사용한 절차

#### Phase 1 — 좌표계 합의
1. 가장 먼저 **원점·축 방향·단위·벽 두께**를 사람이 정의해서 코드에 박음
   (`LR_W`, `LR_D`, `WALL_THICKNESS`, `T2`).
2. 거실 하나만 그리는 최소 모델을 만들어 평면도/3D가 같은 데이터를 공유하는 구조 검증.
3. 이 시점에 **"내측면 vs 외측면"** 용어 합의. 이후 모든 채팅에서 이 용어를 고정.

#### Phase 2 — 인접 공간 한 개씩 추가
4. 안방 → 거실의 *공유벽* 한 장만 먼저 정의 (`-T2` 라인).
5. 안방 4면을 차례로: 좌측벽 → 상단벽(도어 분할 포함) → 하단벽(창문 분할) → (공유벽은 4번에서 완료).
6. **벽 하나를 추가할 때마다 평면도에서 끊김/겹침 확인** 후 다음 벽.
7. 도어와 창은 항상 *벽을 잘라낸 부분벽 + Opening 데이터* 두 개를 함께 추가.

#### Phase 3 — 공간 단위 반복
8. 한 공간(예: 안방욕실) 단위로 다음 절차 반복:
   a. 내측 mm 치수 받아쓰기 (주석에 mm 그대로 남김)
   b. 새 파생 상수를 *기존 상수 + 거리*로 정의
   c. 4면 벽 차례로 추가 (도어/창문은 분할)
   d. `Room` 한 줄 추가 (바닥 색/타일)
   e. 평면도에서 시각 확인
   f. 3D에서 시각 확인
9. 이 사이클을 **욕실 → 메인욕실 → 아기방 → 세탁실 → 베란다 → 작업실 → 현관 → 주방 → 복도** 순으로 반복.

#### Phase 4 — 평면도 비교 & 수정
10. 한 번 전체가 잡힌 후, 레퍼런스 도면과 비교 (`docs/floor_plan001.md`).
11. 좌우/상하 반전을 발견 → **Z축 일괄 반전**을 한 번 수행.
    이 때도 컴포넌트가 아니라 *데이터의 좌표계 정의* 수준에서 수정.

#### Phase 5 — 디테일 레이어
12. 상인방(lintel) — 모든 도어 위 100mm 막는 부분벽 일괄 추가
13. 단차벽 — 욕실/현관 단내림 경계 10mm 두께 벽
14. 붙박이장
15. 다운라이트 그룹 (방별)
16. 텍스처 (마루, 타일, 실크, 호두)
17. gltf 가구 (침대, 변기, 세면대, 부엌)
18. 워크스루 충돌 / 도어 인터랙션
19. 야간 모드 / 도시 배경

### 6.3 지켜진 규약 (Claude 작업 지침)

- **한 번에 한 벽**을 원칙으로 한다. 무리하면 누적 오차가 생긴다.
- 새 상수는 항상 기존 상수와의 *상대 거리*로 정의한다.
- mm 원본을 주석에 남긴다 (`// 1574×1233`).
- 좌표를 의심스러우면 평면도부터 본다. 평면도가 맞으면 3D는 자동으로 맞는다.
- 데이터 파일이 길어지는 것은 정상이다 — 한 곳에 모여 있어야 추적 가능.
- 컴포넌트에는 새 형상을 넣지 않는다. 형상은 무조건 `apartment.ts`.
- "fix" 커밋이 6개 있는 이유: 매 단계마다 확인 후 작은 단위로 커밋했기 때문.

### 6.4 이런 워크플로우의 장점

- **롤백이 항상 가능**: 한 벽 단위라 직전 상태로 복구하기 쉽다.
- **AI가 폭주하지 않는다**: 한 턴 작업량이 작아 컨텍스트가 짧고 오류가 적다.
- **시각 검증이 즉시**: 평면도가 항상 살아 있어 잘못된 좌표가 곧바로 보인다.
- **단일 데이터 소스**: 평면도/조감도/워크스루가 자동으로 동기화 → 검증 비용 절감.

## 7. 기술 스택

| 영역 | 선택 |
|---|---|
| 빌드 | Vite 8 + React 19 + TypeScript 5.9 |
| 3D | three 0.183, @react-three/fiber 9, @react-three/drei 10 |
| 라우팅 | @tanstack/react-router 1 (file-based, `routeTree.gen.ts` 자동 생성) |
| 패키지 매니저 | pnpm |
| Node | >=22 |

### 주요 three.js 사용 패턴
- `boxGeometry` + `meshStandardMaterial` 위주, 셰이더 커스텀 없음
- 텍스처는 `useLoader(TextureLoader)` 한 번 로드 → `clone()`해서 방마다 repeat 다르게 설정
- 다운라이트는 `SpotLight` + 디스크 메시 (램프 본체 시각화)
- `RectAreaLightUniformsLib.init()` 한 번 호출 (창문 일광 등 면광원용)
- gltf 모델은 `useGLTF` (drei) 로 lazy 로딩, `<Suspense>`로 감쌈
- 야경 도시 배경: 외벽 바깥쪽에 큰 plane 4장 (북/남 × 낮/밤), 파일 없으면 조용히 skip

### 성능 메모
- 워크스루는 `invalidate()` 기반의 on-demand 렌더 — 마우스/키 입력 시에만 재렌더
- 조명 그룹 토글로 활성 spotlight 수 최소화
- React Compiler는 사용하지 않음 (vite 빌드 속도 우선)

## 9. 새 작업을 받았을 때 체크리스트 (미래 AI용)

1. 변경하려는 것이 **형상**인가, **표시**인가?
   - 형상이면 → `src/data/apartment.ts` 만 건드리면 끝나는 경우가 대부분.
   - 표시(색/라벨/UI)면 → 해당 View 컴포넌트.
2. 좌표를 새로 만들 때 **기존 상수에서 파생**할 수 있는지 먼저 본다.
3. 벽을 추가할 땐 **창/문 분할까지 한 세트로** 추가 (부분벽 + Opening + 상인방).
4. 작업 후 항상 `/` 평면도부터 확인. 평면도가 맞으면 3D는 거의 항상 맞는다.
5. mm 원본을 주석으로 같이 넣어준다. 미래의 자신이 다시 검증해야 할 수 있다.
6. 쪼개기. 한 번에 한 공간, 한 번에 한 벽. 사람이 확인 가능한 단위로 멈춘다.
